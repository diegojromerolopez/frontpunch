## Objective

Prevent the execution of stale or obsolete tasks by defining expiration limits, automatically discarding tasks that remain in queues beyond their configured TTL.

---

## User Story

As a developer,
I want to specify an expiration time for certain tasks (e.g. `@frontpunch.task(expires_in=3600)`),
So that stale, outdated tasks are discarded without running if worker ingestion lags.

---

## Business Value

- Protects database and external API integrations from processing irrelevant backlog data (e.g., executing expired time-sensitive notifications).
- Prevents queue execution cascades where workers spend valuable time executing stale tasks instead of processing fresh workloads.
- Controls resource spending under spike conditions.

---

## Context

### Existing Behavior

Tasks enqueued in active lists wait indefinitely. If a worker cluster goes offline for 6 hours, it will process all accumulated, potentially outdated alerts upon reboot.

### Desired Behavior

Developers write:
`@frontpunch.task(expires_in=600)` (10 minutes expiration).
When enqueued, the client records `expires_at = created_at + 600` inside the payload. Before a worker thread executes a popped task, it checks `expires_at` against the current UTC timestamp. If expired, the worker discards it without execution, logging the discard event and updating statistics.

---

## Requirements

### Functional Requirements

#### FR-1
Update `@frontpunch.task` decorator to accept `expires_in` (integer seconds).

#### FR-2
Client must calculate absolute expiration epoch timestamp `expires_at = created_at + expires_in` and store it in `JobPayload`.

#### FR-3
Worker threads must compare `expires_at` to the current time immediately upon popping the task from Redis.

#### FR-4
If expired, the task must be discarded (not executed) and logged as an expired job event.

### Non-Functional Requirements

#### NFR-1 Performance
- Expiry evaluation check must take less than 0.05ms per job fetch.

#### NFR-2 Reliability
- Clock differences between client and worker must be bounded or handled gracefully (recommend NTP sync).

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

---

## Business Rules

### BR-1
If `expires_in` is not defined, tasks must never expire.

### BR-2
Expired jobs are discarded silently by default (not retried). If an option `dead_on_expiry=True` is set, route them to `frontpunch:dead` instead.

---

## User Flow

### Happy Path (Job Executed Before Expiration)

1. Task registered with `expires_in=60`.
2. Job is enqueued.
3. Worker picks up job after 10 seconds.
4. Worker verifies `now` < `expires_at`, and executes task successfully.

### Alternative Flows

#### AF-1 Job Expired and Discarded

1. Task enqueued with `expires_in=60`.
2. Worker cluster is offline for 2 minutes.
3. Worker boots, pops the task.
4. Worker verifies `now` > `expires_at` (expired by 60 seconds).
5. Worker drops the task, writes warning log, and proceeds to the next job.

#### AF-2 Error State

N/A

---

## Acceptance Criteria

### AC-1 Job Expiration Discard

**Given**
a task is configured with `expires_in=10`

**When**
the task is enqueued and a worker thread pops it 15 seconds later

**Then**
the task is dropped without invoking its target function.

---

### AC-2 No Expiration Default

**Given**
a task is configured without `expires_in` settings

**When**
the task is enqueued and popped 1 hour later

**Then**
the task executes normally.

---

## Edge Cases

### EC-1
The task is popped at the exact millisecond boundary of expiration.

Expected behavior:
- If `now >= expires_at`, discard it. Strict inequality/equality rules must be enforced consistently.

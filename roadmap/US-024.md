## Objective

Log daily processed and failed task counters in Valkey/Redis to track historical performance and expose metrics for dashboard graph visualizations.

---

## User Story

As a system operator,
I want the system to record daily processing and failure rates,
So that I can view historical performance graphs on the dashboard.

---

## Business Value

- Facilitates bottleneck detection and system load trend diagnostics.
- Simplifies capacity planning by exposing daily processing volume graphs.
- Helps monitor execution health over longer stretches (e.g., detecting weekday spikes vs. weekend dips).

---

## Context

### Existing Behavior

Only live aggregate counts (totals processed/failed since boot/reset) are tracked. There is no historical recording of daily throughput.

### Desired Behavior

Whenever a task completes successfully, server middleware increments key `frontpunch:stats:processed:YYYY-MM-DD`. If it fails completely, it increments `frontpunch:stats:failed:YYYY-MM-DD`. Endpoints `GET /stats/historical` return data for the last 30 days. The dashboard renders a clean, responsive SVG line chart representing daily volume trends.

---

## Requirements

### Functional Requirements

#### FR-1
Increment daily metrics keys `frontpunch:stats:processed:YYYY-MM-DD` and `frontpunch:stats:failed:YYYY-MM-DD` on job success/failure events.

#### FR-2
Daily metric keys must be configured with a TTL of 90 days (preventing memory leaks).

#### FR-3
Implement REST endpoint `GET /stats/historical` returning daily metrics for the past 30 days.

#### FR-4
Render a lightweight, accessible SVG bar or line chart in the HTML dashboard interface.

### Non-Functional Requirements

#### NFR-1 Performance
- Chart drawing must not introduce external script dependencies (use native SVGs).
- API querying must execute in under 10ms.

#### NFR-2 Reliability
- Failures to increment stats must not crash active worker threads.

#### NFR-3 Accessibility
- The SVG charts must include descriptions and ARIA labels.

#### NFR-4 Security
- N/A

---

## Business Rules

### BR-1
Time boundaries must resolve strictly using UTC timezone representations (`YYYY-MM-DD`).

### BR-2
If no jobs executed on a specific date within the 30-day query range, the response must populate a value of `0` for that date.

---

## User Flow

### Happy Path

1. Worker thread completes a job.
2. Server middleware identifies UTC date `2026-06-24`.
3. Worker increments `frontpunch:stats:processed:2026-06-24` and sets TTL to 90 days.
4. Operator opens dashboard.
5. Client queries `GET /stats/historical` and receives list of daily data.
6. Javascript generates and displays the SVG line chart.

### Alternative Flows

#### AF-1 Empty State

1. User views dashboard in a brand new installation.
2. ZSET stats return no entries for the past 30 days.
3. System shows empty stats placeholder: "No historical execution data available."

#### AF-2 Error State

N/A

---

## Acceptance Criteria

### AC-1 Daily Stat Increment

**Given**
2 jobs execute and complete successfully on UTC date `2026-06-24`

**When**
the user requests `GET /stats/historical`

**Then**
the response includes a data point for `2026-06-24` with processed count matching `2`.

---

### AC-2 Chart Integration

**Given**
historical stats contain data points

**When**
the operator visits the dashboard

**Then**
an SVG element with ID `throughput-chart` is populated and visible on the page.

---

## Edge Cases

### EC-1
Valkey/Redis clock is updated or out of sync (causing logs to write to double dates).

Expected behavior:
- Rely strictly on Redis/Valkey system time (`TIME` command) or local worker timestamps aligned with NTP.

---

## Data Model Impact

### New Entities

#### frontpunch:stats:processed:<YYYY-MM-DD> (String)
- Holds integer count of successful executions on the date.
- TTL: 90 days.

#### frontpunch:stats:failed:<YYYY-MM-DD> (String)
- Holds integer count of fatal failures on the date.
- TTL: 90 days.

### Modified Entities

None

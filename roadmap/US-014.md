## Objective

Protect sensitive data parameters in transit and at rest in the message broker by encrypting specific job arguments in-memory before serialization.

---

## User Story

As a security compliance officer,
I want to specify which job parameters must be encrypted in Redis,
So that sensitive user data (e.g. passwords, SSNs, access tokens) is never stored in plain text on the message broker.

---

## Business Value

- Facilitates regulatory compliance (GDPR, HIPAA, PCI-DSS) for background task parameter processing.
- Restricts database/broker administrators from viewing sensitive user information in Redis keys or backups.
- Mitigates impact of unauthorized broker access breaches.

---

## Context

### Existing Behavior

All task arguments passed to `.delay()` are serialized directly as clear text JSON values within the Valkey/Redis payload lists.

### Desired Behavior

Developers write:
`@frontpunch.task(encrypt=["ssn", "api_key"])`
When `.delay(user_id, ssn_value, api_key_value)` is called, the client encrypts the designated parameter values using a locally configured symmetric key (e.g., AES-GCM) before serialization. The worker daemon decrypts these arguments in-memory immediately prior to task execution.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `@frontpunch.task(encrypt=[...])` parameter list configuration.

#### FR-2
Client must replace plain-text target arguments in `JobPayload` with base64-encoded encrypted cipher envelopes.

#### FR-3
Worker must inspect payload metadata and decrypt encrypted arguments using the local key before function invocation.

#### FR-4
If the decryption key is missing or invalid, the task must fail and raise `DecryptionError`.

### Non-Functional Requirements

#### NFR-1 Performance
- Encryption/decryption must add less than 0.5ms of latency to task execution.

#### NFR-2 Reliability
- Key loading must fail-fast on worker startup if keys are formatted incorrectly.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Encryption keys must never be written, stored, or cached in Valkey/Redis.

---

## Business Rules

### BR-1
The encryption/decryption key must be derived from a local environment variable (e.g. `FRONTPUNCH_ENCRYPTION_KEY`) or in-memory settings.

### BR-2
Logs printed by the client or worker must never display the decrypted values of encrypted arguments.

---

## User Flow

### Happy Path

1. Developer registers task `@frontpunch.task(encrypt=["token"])`.
2. Developer calls `task.delay("user1", "sensitive-token-abc")`.
3. Client encrypts "sensitive-token-abc" using the key, serializes it, and enqueues to Redis.
4. Worker pops job, decrypts "sensitive-token-abc" using the key in-memory, and passes it to the task function.

### Alternative Flows

#### AF-1 Decryption Key Missing on Worker

1. Worker picks up job with encrypted field "token".
2. Worker lacks `FRONTPUNCH_ENCRYPTION_KEY` configuration.
3. Worker raises `DecryptionError`, logs a high-severity error (excluding the ciphertext), and sends job to retries/dead queue.

---

## Acceptance Criteria

### AC-1 Encryption In Transit

**Given**
a task configured with `encrypt=["secret"]` and key configuration is present

**When**
the developer calls `task.delay("normal_arg", "my_secret_val")`

**Then**
the payload enqueued in Redis shows `"normal_arg"` in clear text, but the second argument is a ciphertext string.

---

### AC-2 Decryption Prior to Execution

**Given**
an enqueued task with encrypted arguments

**When**
the worker executes the task with a valid decryption key

**Then**
the task function receives the decrypted plain text string `"my_secret_val"`.

---

## Edge Cases

### EC-1
Decryption key is changed/rotated while encrypted jobs are still pending in the queue.

Expected behavior:
- Decryption fails, raising `DecryptionError`. Key rotation must support multi-key transition configurations.

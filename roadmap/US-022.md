## Objective

Provide API endpoints and UI dashboard controls to clear the dead-letter list (DLQ), bulk-retry all dead jobs, or selectively delete individual dead jobs.

---

## User Story

As an administrator,
I want to clear the entire dead-letter list or select specific dead jobs to delete,
So that I can clean up failed jobs and manage recovery at scale.

---

## Business Value

- Speeds up incident recovery by allowing operators to retry thousands of failed jobs with a single click.
- Prevents broker memory exhaustion by purging obsolete, unrecoverable tasks from Redis.
- Streamlines operational management of large-scale task execution failures.

---

## Context

### Existing Behavior

Only single job retries (`POST /queues/dead/retry` for one JID) are supported. Cleaning up or bulk retrying requires running low-level Valkey/Redis CLI commands.

### Desired Behavior

Expose three REST API endpoints:
- `POST /queues/dead/retry_all` (re-queues all dead jobs).
- `DELETE /queues/dead` (deletes all dead jobs).
- `DELETE /queues/dead/{jid}` (deletes a specific dead job).
The dashboard UI displays "Retry All" and "Clear All" buttons, along with a "Delete" button next to each dead task.

---

## Requirements

### Functional Requirements

#### FR-1
Implement REST endpoint `POST /queues/dead/retry_all` to atomically or chunk-by-chunk move all jobs from `frontpunch:dead` back to their active queue lists.

#### FR-2
Implement `DELETE /queues/dead` to clear the entire dead-letter queue list.

#### FR-3
Implement `DELETE /queues/dead/{jid}` to delete a specific dead job matching the JID.

#### FR-4
Expose these actions as buttons in the HTML dashboard interface.

### Non-Functional Requirements

#### NFR-1 Performance
- Bulk actions on up to 10,000 tasks must execute in under 200ms using pipelined commands.

#### NFR-2 Reliability
- Actions must run inside transaction blocks to prevent partial-state corruption.

#### NFR-3 Accessibility
- Action buttons must include confirmation dialogs (preventing accidental purges).

#### NFR-4 Security
- Restrict bulk operations via the configured dashboard authentication token.

---

## Business Rules

### BR-1
Retrying dead jobs must reset their retry counts to 0.

### BR-2
Bulk operations must return the count of affected jobs.

---

## User Flow

### Happy Path (Retry All)

1. Operator clicks "Retry All" on the dashboard.
2. Browser shows a confirmation dialog.
3. Operator confirms; client calls `POST /queues/dead/retry_all`.
4. API moves 50 dead jobs back to active lists and returns count (50).
5. Dashboard refreshes to show 0 dead jobs and displays a success toast.

### Alternative Flows

#### AF-1 Delete Specific Job

1. Operator views dead job panel.
2. Clicks "Delete" icon next to JID `job-123`.
3. Client sends `DELETE /queues/dead/job-123`.
4. API deletes the job from list, and the dashboard removes it from the screen.

#### AF-2 Error State

N/A

---

## Acceptance Criteria

### AC-1 Bulk Retry All

**Given**
the dead list `frontpunch:dead` contains 5 tasks

**When**
the operator triggers `POST /queues/dead/retry_all`

**Then**
all 5 tasks are moved to active queues, the dead list is empty, and response returns `{"retried": 5}`.

---

### AC-2 Clear DLQ

**Given**
the dead list `frontpunch:dead` contains 5 tasks

**When**
the operator triggers `DELETE /queues/dead`

**Then**
all 5 tasks are deleted from Redis, the list is empty, and response returns `{"deleted": 5}`.

---

## Edge Cases

### EC-1
Operator clicks bulk action when the DLQ is empty.

Expected behavior:
- Return HTTP 200 with `{"retried": 0}` or `{"deleted": 0}` and display message "No jobs in dead-letter queue."

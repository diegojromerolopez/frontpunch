## Objective

Developers must be able to group multiple parallel jobs into a single Batch workflow, executing a completion callback task automatically once all batch items execute successfully.

---

## User Story

As a Python developer,
I want to group jobs into a `Batch` context manager and specify an `on_complete` callback task,
So that I can trigger downstream actions automatically when a set of parallel tasks completes.

---

## Business Value

- Unlocks complex workflow modeling (MapReduce patterns, parallel processing merges).
- Improves coordination efficiency without busy-waiting threads.
- Enables multi-step batch operations (e.g. charge billing then send aggregate invoice).

---

## Context

### Existing Behavior

Jobs run independently; coordinating a final completion action requires polling states or manual chain links.

### Desired Behavior

A developer initializes `batch = frontpunch.Batch(description="billing-batch")`. Inside the `with batch:` context manager, calling `.delay()` registers tasks to the batch. Batch metadata is stored in Redis. As tasks complete, server middleware decrements the pending batch count. When it hits 0, the callback task is automatically enqueued.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `frontpunch.Batch(description)` object supporting the context manager protocol (`__enter__`/`__exit__`).

#### FR-2
Store batch metadata in a Valkey/Redis Hash (`frontpunch:batches:<batch_id>`) containing total task count, pending count, and callback details.

#### FR-3
Update server execution middleware to automatically detect `batch_id` in job payloads and atomically decrement the pending count (`DECR` / transaction).

#### FR-4
When pending count reaches 0, enqueue the `on_complete` callback task.

### Non-Functional Requirements

#### NFR-1 Performance
- Decrement check and callback enqueuing must occur atomically within 5ms.

#### NFR-2 Reliability
- Batch callback enqueuing must be idempotent to prevent double callbacks on duplicate task completions.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Batch structures must respect standard secure queue permissions.

---

## Business Rules

### BR-1
A batch must only execute its callback if all jobs within the batch succeed. If any job fails, the callback is suspended until the retries either succeed or are exhausted.

---

## User Flow

### Happy Path

1. Developer writes:
   ```python
   batch = frontpunch.Batch()
   batch.on_complete = "send_invoice"
   with batch:
       charge_card.delay(user1)
       charge_card.delay(user2)
   ```
2. Two tasks run in parallel.
3. Task 1 completes, decrements pending to 1.
4. Task 2 completes, decrements pending to 0.
5. `send_invoice` callback is enqueued.

### Alternative Flows

#### AF-1 Empty State

1. Context manager block contains no jobs.
2. System immediately enqueues the `on_complete` callback task.

#### AF-2 Error State

1. Task within batch fails.
2. Callback is suspended, and the task goes to retries.
3. Once retry succeeds, the decrement proceeds.

---

## Acceptance Criteria

### AC-1 Callback Trigger

**Given**
a batch contains 2 jobs and an `on_complete` callback

**When**
both jobs complete successfully

**Then**
the callback task is enqueued.

---

### AC-2 Failure Suspension

**Given**
a batch job fails and is rescheduled for retry

**When**
other jobs complete

**Then**
the callback is not enqueued until the failed job retries and completes successfully.

---

## Edge Cases

### EC-1
Empty batch context manager block.

Expected behavior:
- Immediately enqueue callback or raise a ValueError.

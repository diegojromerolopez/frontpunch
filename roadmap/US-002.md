## Objective

The job system needs a daemonized CLI process capable of launching a concurrent thread pool that pulls jobs from prioritised queues in Valkey/Redis and executes them.

---

## User Story

As a system operator,
I want to run `frontpunch worker` specifying queues and concurrency,
So that background tasks are processed efficiently in parallel.

---

## Business Value

- Allows horizontal scaling of processing resources.
- Prioritizes critical queues over standard ones.
- Provides thread pool isolation for task executing environments.

---

## Context

### Existing Behavior

No processing daemon exists; tasks remain indefinitely in Valkey/Redis lists.

### Desired Behavior

An operator runs `frontpunch worker --queues default,critical --concurrency 5`. The worker starts a thread pool of size 5. Threads continuously issue blocking `BRPOP` commands on the prioritized queue keys in order (from left to right: `frontpunch:queue:critical` then `frontpunch:queue:default`), deserializes payloads, imports the modules, and runs the functions.

---

## Requirements

### Functional Requirements

#### FR-1
CLI command `frontpunch worker` using `click` or `argparse`.

#### FR-2
Implement a ThreadPoolExecutor wrapper managing `concurrency` worker threads.

#### FR-3
Threads issue blocking `BRPOP` against queue lists, respecting priority order.

#### FR-4
Import path string from payload (e.g. `path.to.my_func`) and execute target function with deserialized arguments.

### Non-Functional Requirements

#### NFR-1 Performance
- Job retrieval and thread dispatch overhead must be under 1ms.

#### NFR-2 Reliability
- Thread crashes must not take down the main worker process daemon.

#### NFR-3 Accessibility
- N/A (CLI daemon)

#### NFR-4 Security
- Execution sandbox must only access registered path imports.

---

## Business Rules

### BR-1
Queues must be prioritized strictly from left-to-right (e.g. `--queues critical,default` will drain `critical` completely before fetching any job from `default`).

### BR-2
A worker thread must be returned to the pool immediately upon completion of task execution.

---

## User Flow

### Happy Path

1. Operator runs `frontpunch worker --queues default --concurrency 2`.
2. Daemon boots up, spawns 2 worker threads.
3. Thread blocks on `BRPOP frontpunch:queue:default 1`.
4. Job arrives, thread pops payload, deserializes it, executes task successfully, and returns to pool.

### Alternative Flows

#### AF-1 Worker Start Failures

1. Operator starts worker with concurrency <= 0.
2. CLI validates argument, raises a usage error, and aborts before booting the pool.

---

## Acceptance Criteria

### AC-1 Prioritized Execution

**Given**
a queue state where `frontpunch:queue:critical` has 1 job and `frontpunch:queue:default` has 1 job

**When**
a worker starts with `--queues critical,default`

**Then**
the job in `critical` is popped and processed first.

---

## Edge Cases

### EC-1
Job payload contains invalid JSON or lacks mandatory schema fields.

Expected behavior:
- Discard job, log severe serialization error, and keep thread alive.

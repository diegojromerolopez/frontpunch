## Objective

Allow developers to configure custom retry backoff strategies or static delays per task, overriding the global default exponential backoff formula.

---

## User Story

As a developer,
I want to specify a custom backoff strategy or static delay for specific tasks,
So that I can control retry intervals in accordance with external service requirements (e.g. static 10-minute wait).

---

## Business Value

- Allows fine-tuned rate-limiting recovery per downstream API client requirements.
- Prevents database or system hammering on task failure loops.
- Grants developers complete control over failure-state behaviors of individual workloads.

---

## Context

### Existing Behavior

US-004 enforces a single, hardcoded default backoff formula: `delay = 15 + (retry_count * 10) + (retry_count^4)` seconds for all failing jobs.

### Desired Behavior

Developers configure tasks:
- `@frontpunch.task(backoff=300)` (static 5 minutes delay).
- `@frontpunch.task(backoff=lambda retry: 10 * (2 ** retry))` (custom backoff callback).
When a task fails, the worker checks its configuration. If a custom backoff is specified, the worker executes/evaluates it to calculate the delay, and schedules the retry accordingly.

---

## Requirements

### Functional Requirements

#### FR-1
Update `@frontpunch.task` decorator to accept a `backoff` parameter (int, float, callable, or string path to callable).

#### FR-2
Evaluate the custom `backoff` setting inside worker thread exception catch blocks to calculate the delay.

#### FR-3
If the custom `backoff` is a callable path string (e.g., `"path.to.my_backoff"`), import it dynamically at runtime.

### Non-Functional Requirements

#### NFR-1 Performance
- Custom backoff calculation must run in under 0.1ms.

#### NFR-2 Reliability
- If the custom backoff calculation raises an exception or returns a non-numeric value, the worker must fall back to the default backoff formula to prevent task drop.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

---

## Business Rules

### BR-1
Custom backoff delay return values must be converted to positive integers representing seconds.

---

## User Flow

### Happy Path

1. Developer registers `@frontpunch.task(backoff=60)`.
2. Job executes and fails.
3. Worker catches exception, reads static `backoff` configuration (60).
4. Worker schedules task in `frontpunch:scheduled` with score = `now + 60`.

### Alternative Flows

#### AF-1 Custom Callable Evaluates

1. Developer registers `@frontpunch.task(backoff=my_custom_func)`.
2. Job fails at retry 2.
3. Worker calls `my_custom_func(2)`, which returns 120.
4. Worker schedules task with score = `now + 120`.

#### AF-2 Error Fallback

1. Custom callable raises `ZeroDivisionError` due to a bug.
2. Worker catches the error, logs a warning, and falls back to calculating default exponential backoff (15 + (retry_count * 10) + (retry_count^4)).

---

## Acceptance Criteria

### AC-1 Static Custom Backoff

**Given**
a task is configured with `backoff=120`

**When**
the task executes and fails

**Then**
the job is rescheduled in `frontpunch:scheduled` with a delay of exactly 120 seconds.

---

### AC-2 Callable Custom Backoff

**Given**
a task is configured with a callable that returns `retry * 10`

**When**
the task fails at retry count 3

**Then**
the job is rescheduled in `frontpunch:scheduled` with a delay of exactly 30 seconds.

---

## Edge Cases

### EC-1
Custom backoff returns a negative number.

Expected behavior:
- Fall back to the default exponential delay.

## Objective

Prevent overloading third-party systems and API targets by implementing global concurrent and sliding window rate limiters in task execution context blocks.

---

## User Story

As a Python developer,
I want to wrap blocks of code with a rate limiter context manager inside my jobs,
So that I do not overload third-party services or exceed external API rate limits globally across worker instances.

---

## Business Value

- Protects third-party API quotas and prevents rate-limit lockouts.
- Ensures compliance with concurrency constraints of downstream partners.
- Prevents database connection exhaustion by placing concurrent block throttles on heavy jobs.

---

## Context

### Existing Behavior

Tasks execute as fast as threads can pull them. If 50 workers run concurrently, they can issue 50 parallel requests to a single external service, causing rate limit blockages.

### Desired Behavior

Developers write context manager blocks:
- `with frontpunch.limiter.concurrent("stripe-api", limit=5):`
- `with frontpunch.limiter.window("stripe-api", limit=100, interval=60):`
If the limit is exceeded globally, the context block raises a `RateLimitExceeded` exception, allowing the task worker to reschedule the job for later execution.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `frontpunch.limiter.concurrent(name, limit)` using a Redis-backed token/semaphore list.

#### FR-2
Implement `frontpunch.limiter.window(name, limit, interval)` using a Redis Sorted Set (ZSET) sliding window log.

#### FR-3
Limiter contexts must raise `frontpunch.exceptions.RateLimitExceeded` if bounds are breached.

#### FR-4
Limiter locks must be shared globally across all worker daemons via Redis.

### Non-Functional Requirements

#### NFR-1 Performance
- Rate limit checks must run atomically in less than 2ms using Redis Lua scripting.

#### NFR-2 Reliability
- Limiter keys must have TTL values to prevent permanent lockout states if workers crash during context execution.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

---

## Business Rules

### BR-1
Concurrent locks must be released atomically when leaving the context manager block (success or fail).

### BR-2
Sliding window limits must remove entries older than the window interval before performing checks.

---

## User Flow

### Happy Path

1. Job enters concurrent rate limiter context block.
2. Redis check reports count (2) is less than limit (5).
3. Job executes external API call.
4. Job exits block, releasing the concurrent token in Redis.

### Alternative Flows

#### AF-1 Limit Exceeded

1. Job enters concurrent rate limiter block.
2. Redis reports count (5) is equal to limit (5).
3. Context block raises `RateLimitExceeded` exception.
4. Exception is caught by task worker, rescheduling the job with backoff.

#### AF-2 Error State

N/A

---

## Acceptance Criteria

### AC-1 Concurrent Limit

**Given**
a concurrent rate limiter for `stripe-api` configured with limit 2

**When**
3 jobs try to enter the context block simultaneously

**Then**
2 jobs execute successfully and 1 job raises a `RateLimitExceeded` exception.

---

### AC-2 Sliding Window Limit

**Given**
a window rate limiter for `mailgun` configured with limit 10 per 60 seconds

**When**
11 jobs try to enter the block within the same 60-second window

**Then**
10 jobs execute successfully and the 11th job raises `RateLimitExceeded`.

---

## Edge Cases

### EC-1
Worker process crashes while holding a concurrent lock.

Expected behavior:
- Limiter lock expires automatically after a configurable TTL (e.g. 5 minutes) to prevent permanent deadlock.

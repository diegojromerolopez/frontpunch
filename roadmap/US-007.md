## Objective

Active worker daemons must register themselves and report periodic heartbeats containing runtime execution details, allowing central monitoring systems to track pool health.

---

## User Story

As a DevOps engineer,
I want worker processes to write live heartbeats to Redis,
So that I can monitor active daemons, concurrency limits, and current active jobs centrally.

---

## Business Value

- Provides real-time visibility into processing cluster sizes.
- Enables detection of dead or unresponsive worker processes.
- Serves as the data backbone for dashboard monitoring UI.

---

## Context

### Existing Behavior

Workers run completely isolated; there is no way to know how many workers are active or what they are currently executing.

### Desired Behavior

On startup, each worker registers a unique key in Valkey/Redis (`frontpunch:processes:<hostname>:<pid>`) containing process details (hostname, PID, concurrency, started_at). A background thread inside the daemon updates a heartbeat timestamp in this hash key every 10 seconds, maintaining a TTL of 60 seconds. On clean shutdown, the worker deletes its registry key.

---

## Requirements

### Functional Requirements

#### FR-1
Register process details on worker boot using a Valkey/Redis Hash.

#### FR-2
Start a lightweight background heartbeat thread updating the key's TTL to 60 seconds every 10 seconds.

#### FR-3
Include active task payload details (what jobs are currently executing in threads) in the registry hash data.

#### FR-4
Remove the process key from Valkey/Redis during graceful shutdown.

### Non-Functional Requirements

#### NFR-1 Performance
- Heartbeat updates must use lightweight writes to not overload Valkey/Redis network I/O.

#### NFR-2 Reliability
- Heartbeat thread failure must log warnings but not crash task-processing pools.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Hostnames must be resolved safely without exposing internal IP routing configurations.

---

## Business Rules

### BR-1
Dead process detection: Any process key that has not updated its heartbeat within 60 seconds will naturally expire due to TTL settings.

### BR-2
Heartbeats must compile current running jobs dynamically using a thread-safe tracker.

---

## User Flow

### Happy Path

1. Operator runs `frontpunch worker --concurrency 5`.
2. Daemon generates key `frontpunch:processes:worker-node-1:3485`.
3. Daemon writes metadata (PID, concurrency, started_at) to Redis.
4. Heartbeat thread starts.
5. Every 10 seconds, heartbeat thread refreshes the TTL of the key to 60 seconds.
6. Daemon shuts down, key is deleted.

### Alternative Flows

#### AF-1 Empty State

N/A

#### AF-2 Error State

1. Heartbeat thread encounters a Valkey connection error.
2. Thread catches the connection exception, logs a warning, and retries in the next cycle.

---

## Acceptance Criteria

### AC-1 Process Registration

**Given**
a worker process starts successfully

**When**
the boot sequence completes

**Then**
a Hash key `frontpunch:processes:<hostname>:<pid>` exists in Redis with a TTL of 60 seconds.

---

### AC-2 Heartbeat Refresh

**Given**
a worker process is running normally

**When**
30 seconds pass

**Then**
the TTL of the process registry key remains refreshed (between 50 and 60 seconds).

---

## Edge Cases

### EC-1
Worker process crashes abruptly (SIGKILL).

Expected behavior:
- Key naturally expires and disappears from Redis after 60 seconds.

## Implementation Notes

Wrap the heartbeat thread logic in try-except blocks to prevent crashes from propagating. Use absolute UTC epoch times for started_at.

---

## AI Instructions

1. Use `threading.Thread(daemon=True)` for the heartbeat loop.
2. Recover gracefully if Valkey becomes temporarily unreachable.

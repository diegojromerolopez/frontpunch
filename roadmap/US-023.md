## Objective

Provide API endpoints and dashboard interfaces to search, inspect, and delete future-scheduled tasks or failed jobs waiting for backoff retries.

---

## User Story

As an operator,
I want to view and delete jobs currently waiting in the scheduled queue,
So that I can cancel future executions or troubleshoot tasks waiting for retry.

---

## Business Value

- Enables cancellation of mistakenly scheduled resource-heavy tasks before they run.
- Simplifies debugging of failed workflows by letting developers inspect tasks currently waiting in backoff retries.
- Reduces administration overhead by bringing parity to queue inspect operations.

---

## Context

### Existing Behavior

Tasks in the `frontpunch:scheduled` Sorted Set are invisible to the dashboard and cannot be cancelled or removed without using low-level Redis CLI commands.

### Desired Behavior

Expose REST endpoints:
- `GET /queues/scheduled?page=1&limit=50` (returns a paginated list of scheduled/retry tasks).
- `DELETE /queues/scheduled/{jid}` (deletes a specific task from the scheduled set).
The dashboard UI features a "Scheduled & Retries" tab containing a table of waiting tasks with their targets, error histories, and "Cancel" buttons.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `GET /queues/scheduled` returning a paginated list of task payloads, execution target timestamps, and JIDs.

#### FR-2
Implement `DELETE /queues/scheduled/{jid}` to search and remove the matching task payload from the `frontpunch:scheduled` Sorted Set.

#### FR-3
Expose a "Scheduled Tasks" table and cancellation actions in the HTML dashboard.

### Non-Functional Requirements

#### NFR-1 Performance
- Paginated retrieval must execute in under 20ms using `ZRANGE` or `ZSCAN`.

#### NFR-2 Reliability
- Deleting a scheduled task must atomically release any unique job locks (US-010) tied to its JID.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Restrict endpoint operations via the configured dashboard authorization token.

---

## Business Rules

### BR-1
Tasks retrieved from `frontpunch:scheduled` must be sorted by their scheduled execution time in ascending order.

### BR-2
Deletion operations must use atomic Lua scripts to locate the payload containing the JID inside the ZSET and delete it.

---

## User Flow

### Happy Path

1. Operator navigates to the "Scheduled" tab on the dashboard.
2. JavaScript calls `GET /queues/scheduled`.
3. Table renders tasks showing execution targets (e.g. "Runs in 10 minutes").
4. Operator clicks "Cancel" next to `job-999`.
5. Client issues `DELETE /queues/scheduled/job-999`.
6. API removes the task from the Sorted Set, and the dashboard removes the row.

### Alternative Flows

#### AF-1 Empty State

1. User views "Scheduled" tab.
2. ZSET contains 0 members.
3. Dashboard displays: "No scheduled or retrying jobs found."

#### AF-2 Error State

N/A

## Acceptance Criteria

### AC-1 Paginated Retrieval

**Given**
the scheduled queue contains 3 tasks

**When**
the operator calls `GET /queues/scheduled`

**Then**
the API returns a list of 3 task payloads sorted by score, including target timestamps.

---

### AC-2 Scheduled Job Deletion

**Given**
a task with JID `job-777` exists in the scheduled queue

**When**
the operator triggers `DELETE /queues/scheduled/job-777`

**Then**
the job is removed from `frontpunch:scheduled` and HTTP 200 is returned.

---

## Edge Cases

### EC-1
An operator attempts to delete a job just as it matures and is moved by the scheduler thread to an active queue.

Expected behavior:
- Return HTTP 404 Not Found since the job is no longer present in the scheduled ZSET.

## API Changes

### Endpoints

#### GET /queues/scheduled?page=1&limit=50
- Response:
  ```json
  {
    "total": 1,
    "jobs": [
      {
        "jid": "job-777",
        "class": "path.to.task",
        "args": [1, 2],
        "queue": "default",
        "run_at": "2026-06-24T17:00:00Z"
      }
    ]
  }
  ```

#### DELETE /queues/scheduled/{jid}
- Response: `{"status": "success", "deleted": "job-777"}`

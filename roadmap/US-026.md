## Objective

Expose dashboard interfaces and API endpoints to dynamically view, update, and manage global rate limit thresholds at runtime without modifying application code or restarting workers.

---

## User Story

As an operator,
I want to dynamically adjust rate limits and throttle limits from the dashboard,
So that I can respond to API changes or external outages in real-time.

---

## Business Value

- Increases operational agility: operators can immediately lower limits if a third-party partner experiences degradation, or raise them when upgraded.
- Eliminates code deployment cycles for trivial threshold updates.
- Centralizes throttling configurations in a single browser-accessible control board.

---

## Context

### Existing Behavior

Rate limits (US-012) are defined statically in Python code blocks (`with frontpunch.limiter.concurrent("name", limit=X):`). Changing limits requires code commits and deployments.

### Desired Behavior

The rate-limiting context manager queries Valkey/Redis (`frontpunch:limiters:config:{name}`) before applying limits. If a dynamic override exists, it uses the override value instead of the code-default limit. The REST API exposes GET and PUT endpoints for limiters, and the dashboard features a "Rate Limiters" control panel.

---

## Requirements

### Functional Requirements

#### FR-1
Rate-limiter checks (US-012) must query `frontpunch:limiters:config:{name}` in Redis and apply overrides.

#### FR-2
Implement `GET /limiters` returning all active rate limit configurations and their current overrides.

#### FR-3
Implement `PUT /limiters/{name}` to dynamically write or update override values (concurrency limit, window limit, window interval) in Redis.

#### FR-4
Expose a Throttle Control panel in the HTML dashboard containing input forms to update limits.

### Non-Functional Requirements

#### NFR-1 Performance
- Dynamic checks must use local in-memory caching (e.g. 5 seconds) to prevent issuing lookup commands to Valkey on every single limiter context enter.

#### NFR-2 Reliability
- If Redis is offline, fall back immediately to the code-configured default limit.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Restrict `PUT /limiters/{name}` endpoints using token authentication rules.

---

## Business Rules

### BR-1
Dynamic override values stored in Redis take absolute precedence over the values declared inside Python task code.

### BR-2
Deleting the config key in Redis must revert the rate limiter back to using the code-defined limits.

---

## User Flow

### Happy Path

1. Operator opens the dashboard "Rate Limiters" tab.
2. User sees `stripe-api` limiter with code-default limit `5`.
3. Operator edits the limit value, setting it to `10`, and saves.
4. Client sends `PUT /limiters/stripe-api` with body `{"limit": 10}`.
5. Workers refresh their local cache and begin allowing up to 10 concurrent jobs.

### Alternative Flows

#### AF-1 Revert to Default

1. Operator clicks "Reset to Code Default" next to `stripe-api` on the dashboard.
2. Client sends a delete override request.
3. Redis key is deleted, and workers fall back to the default limits specified in code.

#### AF-2 Error State

N/A

## Acceptance Criteria

### AC-1 Dynamic Limit Precedence

**Given**
a task is configured in code with concurrent limit `2`, and a Redis override exists with limit `5`

**When**
4 tasks attempt to enter the limiter context concurrently

**Then**
all 4 tasks enter successfully (the limit of 5 is respected).

---

### AC-2 Cache Expiration

**Given**
the worker has cached a dynamic override limit of `5`

**When**
the operator updates the limit to `2` in Redis

**Then**
within 5 seconds, the worker updates its local limit, and subsequent concurrent requests above 2 are throttled.

---

## Edge Cases

### EC-1
The dynamic concurrency limit is updated to a value lower than the number of tasks currently executing inside the block (e.g., limit updated to 2 while 4 tasks are running).

Expected behavior:
- Throttles all subsequent tasks trying to enter the block. Running tasks are allowed to complete.

## API Changes

### Endpoints

#### GET /limiters
- Response:
  ```json
  [
    {
      "name": "stripe-api",
      "type": "concurrent",
      "code_default": 5,
      "current_override": 10
    }
  ]
  ```

#### PUT /limiters/{name}
- Request: `{"limit": 10}`
- Response: `{"status": "success", "name": "stripe-api", "limit": 10}`

## Objective

Jobs must support delayed or future schedules, requiring the ability to enqueue them into a sorted set and periodically move mature jobs to active lists.

---

## User Story

As a Python developer,
I want to schedule jobs to run in the future using `.perform_in()` or `.perform_at()`,
So that execution is deferred to a desired time.

---

## Business Value

- Enables time-based tasks (e.g. reminder emails, weekly reports).
- Evens out load spikes by scheduling heavy processing to off-peak hours.

---

## Context

### Existing Behavior

Tasks are immediately pushed to active lists and executed right away.

### Desired Behavior

Developers call `.perform_in(300, arg1)` (relative seconds) or `.perform_at(datetime_obj, arg1)`. The client serializes the payload and saves it in a Redis Sorted Set named `frontpunch:scheduled` using the target UTC unix timestamp as the score. A scheduler thread inside the worker daemon polls `frontpunch:scheduled`, pops jobs with score <= current timestamp, and pushes them to their active lists.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `task.perform_in(seconds, *args)` and `task.perform_at(datetime, *args)` client methods.

#### FR-2
Scheduled jobs must be saved in Valkey/Redis Sorted Set (`frontpunch:scheduled`) with score = execution epoch timestamp.

#### FR-3
Implement a background daemon scheduler thread inside `frontpunch worker`.

#### FR-4
Scheduler thread must query `frontpunch:scheduled` with `ZRANGEBYSCORE` where score <= now, pop elements atomically, and `LPUSH` them to their active queues.

### Non-Functional Requirements

#### NFR-1 Performance
- Scheduler polling overhead must not impact active task thread throughput.

#### NFR-2 Reliability
- Atomic pops must be enforced using Redis Lua scripting or transactions (`WATCH`/`MULTI`) to prevent duplicate queueing under multiple schedulers.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Scheduled timestamps must resolve to UTC consistently to prevent timezone offset execution bypasses.

---

## Business Rules

### BR-1
Absolute execution times must be converted to UTC unix timestamps before submission.

### BR-2
The scheduler thread must run at a configurable interval (default: 5 seconds).

---

## User Flow

### Happy Path

1. Developer calls `send_reminder.perform_in(10, 456)`.
2. Client saves job in `frontpunch:scheduled` with score = `now + 10`.
3. 10 seconds elapse.
4. Worker scheduler thread detects job, removes it from ZSET, and enqueues to `frontpunch:queue:default`.
5. Worker thread executes the task.

### Alternative Flows

#### AF-1 Empty State

1. Scheduler thread wakes up.
2. Checks `frontpunch:scheduled` and finds no mature elements.
3. Thread goes back to sleep without modifying any queues.

---

## Acceptance Criteria

### AC-1 Scheduled Insertion

**Given**
a decorated task `my_task`

**When**
the developer calls `my_task.perform_in(30, "arg")`

**Then**
the job is added to the `frontpunch:scheduled` Sorted Set with score matching `now + 30`.

---

### AC-2 Scheduler Drain

**Given**
a job exists in `frontpunch:scheduled` with a score less than the current time

**When**
the scheduler cycle runs

**Then**
the job is removed from the sorted set and added to its target active queue.

---

## Edge Cases

### EC-1
Valkey/Redis clock is slightly out of sync with host clock.

Expected behavior:
- Use Redis server time (`TIME` command) or assume host clock is synchronized via NTP.

## Objective

Provide a web-based, graphical dashboard interface to visually monitor queue statistics, track active workers, and manage dead tasks directly in the browser.

---

## User Story

As an operator,
I want to access a web dashboard in my browser,
So that I can visually monitor job statistics, inspect active workers, and retry failed dead jobs without executing command-line commands.

---

## Business Value

- Significantly reduces operational MTTR (Mean Time to Resolution) by providing a one-click manual retry button for failed dead jobs.
- Offers non-technical stakeholders visibility into queue health and processing status.
- Prevents errors caused by developers running raw Redis query commands during incidents.

---

## Context

### Existing Behavior

Only the REST API endpoints (`GET /stats`, `GET /queues`, etc.) exist. Checking system state requires command line `curl` or raw Valkey/Redis CLI statements.

### Desired Behavior

Expose a `/dashboard` HTTP endpoint in the dashboard app (from US-008) that serves a responsive, modern HTML5 web page. The interface uses CSS gradients and clean typography, featuring cards showing total tasks, active queues tables, worker lists, and a dead-letter management terminal where users can select and retry/delete dead tasks.

---

## Requirements

### Functional Requirements

#### FR-1
Expose a `GET /dashboard` endpoint serving a semantic HTML5 template.

#### FR-2
Include a real-time data refreshing mechanism (e.g. client-side fetch polling every 5 seconds).

#### FR-3
Provide interactive buttons to trigger job retries for entries in the dead-letter list.

#### FR-4
Expose queue stats, active workers registry details, and queue length visualizations.

### Non-Functional Requirements

#### NFR-1 Performance
- Initial page payload (HTML + CSS + JS) must load in under 100ms.

#### NFR-2 Reliability
- The interface must degrade gracefully if the underlying Valkey/Redis connection goes offline, showing a clear connection failure status.

#### NFR-3 Accessibility
- Use semantic HTML tags (`<header>`, `<main>`, `<section>`).
- Support keyboard navigation and contrast ratios complying with WCAG 2.1 AA rules.

#### NFR-4 Security
- Dashboard access must support configuration-based HTTP Basic Authentication.

---

## Business Rules

### BR-1
Dashboard interactions must respect the underlying REST API response rules.

### BR-2
Actions (like retrying a job) must show loading indicators during client-side fetch events.

---

## User Flow

### Happy Path

1. User opens browser to `http://localhost:8000/dashboard`.
2. Page loads, CSS renders, and Javascript makes fetch calls to `/stats` and `/queues`.
3. System displays statistics and queue structures.
4. User clicks "Retry" next to a dead job; the page calls the API, shows success, and removes the job from the dead list.

### Alternative Flows

#### AF-1 Empty State

1. User opens the dashboard.
2. No active workers are running, and queues are empty.
3. Dashboard displays "No active workers detected" and "No queued tasks" empty state panels.

#### AF-2 Error State

1. User opens the dashboard when Redis is offline.
2. Dashboard displays a global banner "Valkey/Redis Connection Unavailable. Reconnecting..."

---

## Acceptance Criteria

### AC-1 Dashboard Rendering

**Given**
the dashboard server is active and running

**When**
the user navigates to `/dashboard`

**Then**
an HTML page is served containing elements with IDs `stats-grid`, `queue-table`, and `workers-list`.

---

### AC-2 Interactive Dead Job Retry

**Given**
a dead job is shown in the dead-letter panel

**When**
the user clicks the "Retry" button for that job

**Then**
the client issues a `POST` request to `/queues/dead/retry`, the job is removed from the list, and a success toast is shown.

---

## Edge Cases

### EC-1
User clicks "Retry" on a dead job that has already been retried or deleted by another administrator.

Expected behavior:
- Show warning message: "Job not found. It may have already been retried." and refresh the list.

## UI Requirements

### Components

- StatsCard (Processed, Failed, Dead counts)
- QueueTable (Name, Length columns)
- WorkersList (Host, PID, Concurrency, Active Job detail)
- DeadLetterList (JID, Error Class, Failed At, Actions)

### States

- Loading: Skeleton loaders shown during initial load.
- Loaded: Normal data view.
- Empty: "No items" alerts for workers/dead queues.
- Error: Global error overlay banner.

### Accessibility

- Full keyboard support for tabs and buttons.
- Proper ARIA labels on dynamic action buttons.

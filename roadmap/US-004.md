## Objective

Active tasks that encounter transient failures must be intercepted, checked against retry ceilings, scheduled with exponential backoff delay, or routed to a dead-letter queue.

---

## User Story

As a system engineer,
I want failed tasks to be automatically retried with exponential backoff,
So that transient external system errors (e.g. database lockouts, API rate limits) do not cause task loss.

---

## Business Value

- Increases execution resilience.
- Automates recovering flows, minimizing manual operations.
- Isolates broken tasks in a dead-letter queue for forensic debugging.

---

## Context

### Existing Behavior

If a task raises an exception, the thread execution scope crashes and no recovery is attempted.

### Desired Behavior

When a task fails, the worker catches the exception. If `retry_count < max_retries`, it calculates backoff delay using `15 + (retry_count * 10) + (retry_count^4) seconds`, updates the job payload's `retry_count`, and schedules it by adding it back to `frontpunch:scheduled`. If the retries are exhausted, it pushes the job to a dead list key `frontpunch:dead`.

---

## Requirements

### Functional Requirements

#### FR-1
Catch all non-system exceptions during task execution inside worker threads.

#### FR-2
Implement exponential backoff calculation:
$$\text{delay} = 15 + (\text{retry\_count} \times 10) + (\text{retry\_count}^4) \text{ seconds}$$

#### FR-3
Increment payload `retry_count` and write back to `frontpunch:scheduled` sorted set.

#### FR-4
If `retry_count >= max_retries`, push job payload to Redis List `frontpunch:dead`.

### Non-Functional Requirements

#### NFR-1 Performance
- Delay calculation and schedule insertion must not block active thread pools.

#### NFR-2 Reliability
- Fatal system exceptions (e.g., `KeyboardInterrupt`, `SystemExit`) must bypass catch blocks to allow worker termination.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Payload credentials must remain preserved and encrypted (if configured) during retry states.

---

## Business Rules

### BR-1
Only retry tasks if `max_retries` > 0. If `max_retries` is 0, send failed tasks to `frontpunch:dead` immediately.

### BR-2
Job payload must include error message and exception class details under `error_message` and `error_class` keys during retries.

---

## User Flow

### Happy Path

1. Task executes, raises `HTTPError`.
2. Worker thread intercepts exception, verifies `retry_count` (0) < `max_retries` (5).
3. Backoff is calculated (15 seconds).
4. Task payload is enqueued into `frontpunch:scheduled` with score = `now + 15`.
5. `retry_count` is updated to 1.

### Alternative Flows

#### AF-1 Empty State

N/A (Error flows deal with failed states only)

#### AF-2 Error State

1. Task fails and raises an exception.
2. The retry mechanism fails to connect to Redis/Valkey.
3. System logs severe fatal exception, worker keeps running.

---

## Acceptance Criteria

### AC-1 Successful Retry Scheduling

**Given**
a task raises a standard exception and `retry_count` is 0

**When**
the execution loop catches the error

**Then**
the job is rescheduled in `frontpunch:scheduled` with `retry_count` set to 1.

---

### AC-2 Exhausted Retries Routing

**Given**
a task raises a standard exception and `retry_count` is 5 (equal to `max_retries`)

**When**
the execution loop catches the error

**Then**
the job is pushed to the `frontpunch:dead` queue list.

---

## Edge Cases

### EC-1
Task raises a memory exhaustion error or system exit signals.

Expected behavior:
- Propagation of system signal, bypassing retry mechanism to trigger daemon shutdown.

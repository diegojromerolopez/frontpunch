## Objective

Ensure execution safety by electing a single active scheduler leader among running workers, preventing duplicate cron job enqueues when scaling horizontally.

---

## User Story

As a DevOps engineer scaling my processing cluster,
I want only one scheduler node to execute cron evaluations at any given time,
So that recurring tasks (e.g. daily backups) are never triggered multiple times across separate worker instances.

---

## Business Value

- Guarantees execution safety and data correctness under multi-node worker clusters.
- Prevents resource exhaustion from duplicate processing of heavy periodic operations.
- Simplifies operational overhead by avoiding the configuration of dedicated, single-point-of-failure scheduling nodes.

---

## Context

### Existing Behavior

If multiple workers run the cron scheduler thread (defined in US-013), they will concurrently read the cron schedule list and potentially enqueue duplicate instances of the same due tasks.

### Desired Behavior

Workers use a distributed lock key in Valkey/Redis (`frontpunch:locks:scheduler`) to perform leader election. The worker that acquires the lock becomes the active scheduler leader. It periodically renews the lock. Other workers remain in passive standby mode. If the leader fails to renew, the lock expires, and another worker takes over.

---

## Requirements

### Functional Requirements

#### FR-1
Implement leader election using a Redis string lock acquired via atomic `SET key val NX PX 10000`.

#### FR-2
The elected scheduler leader thread must renew the lock every 5 seconds.

#### FR-3
Standby workers must attempt to acquire the lock if it expires or is released.

#### FR-4
Use atomic transaction checks on task execution records as a secondary guard to prevent double-scheduling.

### Non-Functional Requirements

#### NFR-1 Performance
- Leader election checks must add negligible CPU/Network overhead.

#### NFR-2 Reliability
- Standby takeover must occur within 10 seconds of leader crash.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

---

## Business Rules

### BR-1
Only the elected leader process is authorized to read the cron registry and push matured tasks to active queues.

### BR-2
The leader lock must expire after a maximum TTL of 10 seconds if not renewed.

---

## User Flow

### Happy Path

1. Three worker daemons (A, B, C) start.
2. Worker A succeeds in writing `frontpunch:locks:scheduler` in Redis.
3. Worker A becomes the active scheduler leader.
4. Workers B and C fail to acquire the lock and go to standby sleep.
5. Worker A evaluates schedules, enqueues tasks, and renews the lock every 5 seconds.

### Alternative Flows

#### AF-1 Leader Crash & Standby Takeover

1. Worker A (leader) crashes.
2. 10 seconds elapse; the lock `frontpunch:locks:scheduler` expires in Redis.
3. Worker B attempts to acquire lock and succeeds.
4. Worker B becomes the new scheduler leader and resumes evaluation.

#### AF-2 Error State

N/A

## Acceptance Criteria

### AC-1 Single Active Scheduler

**Given**
3 active worker processes with cron scheduling active

**When**
the scheduler loop executes

**Then**
only 1 worker process successfully acquires the leader lock and performs task enqueuing, while 2 processes remain standby.

---

### AC-2 Standby Takeover

**Given**
Worker A is the leader, and Worker B is standby

**When**
Worker A crashes and the lock expires

**Then**
Worker B acquires the lock and begins executing the scheduler loops.

---

## Edge Cases

### EC-1
Network split-brain where two workers temporarily believe they are the leader.

Expected behavior:
- Use Redis transactional check-and-set updates on the next run timestamp to prevent double-pushing even if multiple nodes perform enqueuing simultaneously.

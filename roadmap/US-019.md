## Objective

Extend the job batch workflow system to support fine-grained post-execution hooks: executing success actions when all batch jobs complete clean, or executing failure callbacks if any job dies.

---

## User Story

As a Python developer,
I want to register success and failure callbacks on a job batch,
So that I can trigger specific recovery pipelines if any batch job fails completely, or proceed with cleanups only when all succeed.

---

## Business Value

- Supports automatic rollback routines for complex workflows (e.g. refunding successful charges if a later step in the batch fails).
- Minimizes operational overhead by automating incident management for batches.
- Unlocks sophisticated transactional modeling across distributed background workers.

---

## Context

### Existing Behavior

Only the `on_complete` callback is supported. It triggers once the last job in the batch finishes executing, regardless of whether any job failed or was moved to the dead list.

### Desired Behavior

Developers write:
```python
batch = frontpunch.Batch()
batch.on_success = "path.to.finalize_order"
batch.on_death = "path.to.cancel_reservations"
```
The batch tracking metadata in Redis tracks the failure status of the batch. If a batch job exhausts all retries and is moved to `frontpunch:dead`, the batch is flagged as failed. Once the pending count hits 0:
- If no failures occurred, `on_success` is enqueued.
- If any job died, `on_death` is enqueued.

---

## Requirements

### Functional Requirements

#### FR-1
Add `on_success` and `on_death` string properties to the `Batch` class.

#### FR-2
Store success/death callback metadata in the Redis Hash `frontpunch:batches:<batch_id>`.

#### FR-3
Update server execution middleware: if a batch task exhausts retries and is pushed to `frontpunch:dead`, decrement the pending count and set the batch's `failed` field to `1`.

#### FR-4
When the pending count reaches 0, atomically evaluate the `failed` flag, and enqueue either the `on_success` or the `on_death` callback task.

### Non-Functional Requirements

#### NFR-1 Performance
- Callback evaluations must run atomically inside Redis in under 5ms.

#### NFR-2 Reliability
- Multiple concurrent completions must be resolved safely without triggering duplicate callbacks.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

## Business Rules

### BR-1
A batch is considered successful only if all jobs completed without exhaustively failing (zero dead jobs).

### BR-2
A batch is marked failed as soon as any single task in the batch is moved to the dead-letter queue.

---

## User Flow

### Happy Path (All Jobs Succeed)

1. Developer schedules a batch of 2 jobs with `on_success` and `on_death` callbacks.
2. Both jobs execute successfully.
3. The last job decrements pending count to 0.
4. System checks `failed` flag (0).
5. The `on_success` callback task is enqueued.

### Alternative Flows

#### AF-1 Job Death Trigger

1. Developer schedules batch.
2. Job 1 succeeds. Job 2 fails repeatedly, exhausting all 5 retries.
3. Job 2 goes to the dead list.
4. System decrements pending count to 0 and flags batch `failed=1`.
5. The `on_death` callback task is enqueued.

#### AF-2 Error State

N/A

---

## Acceptance Criteria

### AC-1 Success Callback Trigger

**Given**
a batch contains 2 jobs and both `on_success` and `on_death` callbacks

**When**
both jobs execute and finish successfully

**Then**
the `on_success` callback is enqueued and `on_death` is ignored.

---

### AC-2 Death Callback Trigger

**Given**
a batch contains 2 jobs and both `on_success` and `on_death` callbacks

**When**
1 job succeeds and 1 job exhausts retries and goes to the dead queue

**Then**
the `on_death` callback is enqueued and `on_success` is ignored.

---

## Edge Cases

### EC-1
Multiple jobs within the same batch exhaust retries and die.

Expected behavior:
- Trigger `on_death` exactly once when the last job finishes (when pending count reaches 0).

## Objective

Enable developers to register background tasks that execute on a recurring periodic schedule using standard cron expressions.

---

## User Story

As a developer/operator,
I want to register a task to run periodically using cron syntax (e.g. `0 * * * *`),
So that I can automate recurring maintenance, cleanups, or sync scripts.

---

## Business Value

- Eliminates dependency on system-level cron daemons (like `crontab`).
- Co-locates background task definition and execution scheduling in Python application code.
- Ensures cluster-wide execution of periodic tasks without duplicate runs.

---

## Context

### Existing Behavior

Only relative delays (`perform_in`) and one-shot absolute schedulers (`perform_at`) are supported. There is no engine to recurrently reschedule a job.

### Desired Behavior

Developers write:
`frontpunch.cron.add("hourly-cleanup", "0 * * * *", run_cleanup)`
The scheduler thread inside the worker daemon periodically checks registered cron jobs. It calculates the next execution time and enqueues the job into its active list when the current UTC timestamp matches or exceeds the scheduled run time, immediately scheduling the subsequent execution.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `frontpunch.cron.add(name, cron_expression, task)` API.

#### FR-2
Store cron task schedules in Redis (`frontpunch:cron` Hash).

#### FR-3
Implement scheduler checks in the background worker daemon to evaluate cron expressions and enqueue due tasks.

#### FR-4
Support standard 5-field cron syntax (minute, hour, day of month, month, day of week).

### Non-Functional Requirements

#### NFR-1 Performance
- Cron checks must execute in under 1ms per cycle.

#### NFR-2 Reliability
- If multiple worker nodes run schedules, only one node must enqueue a specific cron task occurrence (idempotent scheduling).

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Scheduled tasks must execute under the standard execution permissions.

---

## Business Rules

### BR-1
Time evaluations must always be calculated using UTC timezones.

### BR-2
If the worker is offline when a cron job was scheduled to run, it should run immediately upon worker startup (catch-up execution), unless configured otherwise.

---

## User Flow

### Happy Path

1. Developer registers `daily-report` task to run daily at midnight (`0 0 * * *`).
2. Scheduler stores the schedule in Redis and sets the next run timestamp.
3. Time reaches midnight UTC.
4. Scheduler detects the job is due, enqueues `daily-report` to `frontpunch:queue:default`, and updates the next run timestamp to the next day.
5. Worker thread processes the job.

### Alternative Flows

#### AF-1 Empty State

N/A

#### AF-2 Error State

1. Developer specifies an invalid cron expression `invalid-cron-str`.
2. System raises `ValueError` immediately during registration, blocking boot.

---

## Acceptance Criteria

### AC-1 Cron Job Enqueueing

**Given**
a task registered with cron expression `*/1 * * * *` (every minute)

**When**
a minute mark passes

**Then**
the job is enqueued to its target list exactly once.

---

## Edge Cases

### EC-1
The cron task takes longer to execute than the scheduling frequency (e.g. job runs for 2 minutes but cron triggers every minute).

Expected behavior:
- Enqueue the next occurrence regardless of whether the previous instance is still running (no execution lock unless specified by unique locks).

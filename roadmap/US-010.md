## Objective

Prevent redundant task enqueues or parallel duplicate executions by checking and acquiring job locks in Valkey/Redis based on task names and arguments.

---

## User Story

As a Python developer,
I want to configure unique constraints on tasks (e.g. `@frontpunch.task(unique="until_executed")`),
So that duplicate jobs with identical arguments are not processed concurrently.

---

## Business Value

- Prevents resource waste from redundant processing.
- Avoids data corruption from race conditions in overlapping jobs.
- Reduces DB load by limiting concurrent identical writes.

---

## Context

### Existing Behavior

If the developer calls a task twice with same arguments in quick succession, both are enqueued and run concurrently, potentially creating race conditions.

### Desired Behavior

When enqueuing a unique task, client calculates key `frontpunch:unique:<hash(name+args)>`. If lock exists, enqueue is skipped. Lock types: `until_executing` (released when job starts) or `until_executed` (released when job finishes).

---

## Requirements

### Functional Requirements

#### FR-1
Implement `@frontpunch.task(unique="until_executing"|"until_executed", unique_expiration=3600)`.

#### FR-2
Client must calculate a deterministic lock key based on class name and arguments: `frontpunch:unique:<hash>`.

#### FR-3
Implement lock acquisition check using Valkey/Redis atomic `SETNX` on enqueuing. If lock fails to acquire, skip enqueuing.

#### FR-4
Release lock key on job execution start (`until_executing`) or completion (`until_executed`) using Lua release script.

### Non-Functional Requirements

#### NFR-1 Performance
- Lock checks must add less than 2ms overhead during enqueuing.

#### NFR-2 Reliability
- Locks must have a TTL (`unique_expiration`) to prevent permanent locks in case of worker crashes.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

---

## Business Rules

### BR-1
Skip enqueuing if lock is active. The `.delay()` call should return the `jid` of the existing locked task or `None` (configurable).

---

## User Flow

### Happy Path

1. Task decorated with `unique="until_executed"`.
2. Developer calls `sync_data.delay(user1)`. Lock key `frontpunch:unique:a7b8` is set with TTL.
3. Job is enqueued.
4. Second call `sync_data.delay(user1)` is issued, lock fails, job is skipped.
5. Job executes, finishes, lock is deleted.

### Alternative Flows

#### AF-1 Empty State

N/A

#### AF-2 Error State

1. Task enqueued with unique constraint.
2. Worker daemon crashes during execution.
3. Lock naturally expires after `unique_expiration` seconds, allowing subsequent jobs.

---

## Acceptance Criteria

### AC-1 Lock Acquisition

**Given**
a task is unique and no duplicate is enqueued

**When**
the developer calls the task

**Then**
the lock is set in Redis and the job is enqueued.

---

### AC-2 Duplicate Prevented

**Given**
a unique task lock is already active in Redis

**When**
the developer enqueues a duplicate job with identical arguments

**Then**
the job is not enqueued.

---

## Edge Cases

### EC-1
Worker process crashes while holding an `until_executed` lock.

Expected behavior:
- Lock expires naturally after `unique_expiration` seconds, allowing subsequent jobs.

## Implementation Notes

## AI Instructions

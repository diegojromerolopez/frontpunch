## Objective

Maximize CPU core utilization and execution reliability by implementing a parent supervisor command (`frontpunch swarm`) that spawns and monitors multiple child worker processes.

---

## User Story

As a system administrator,
I want to run `frontpunch swarm` specifying process count and worker settings,
So that multiple background worker processes are managed on the host automatically by a parent supervisor.

---

## Business Value

- Bypasses Python's Global Interpreter Lock (GIL) by distributing tasks across separate child OS processes, maximizing multi-core throughput.
- Automates crash recovery by restarting terminated child workers.
- Simplifies operational setups by requiring only 1 systemd unit to coordinate multiple workers.

---

## Context

### Existing Behavior

The worker CLI only executes a single process containing a thread pool (US-002). Utilizing multiple cores requires manually launching and supervising multiple independent terminal instances.

### Desired Behavior

An operator executes `frontpunch swarm --processes 4 --concurrency 5 --queues default`. The CLI boots a parent supervisor process. The supervisor spawns 4 separate child processes executing standard worker loops. The parent intercepts signals (SIGINT, SIGTERM, SIGTSTP) and propagates them to children. It monitors child PIDs and spawns replacements if any child terminates unexpectedly.

---

## Requirements

### Functional Requirements

#### FR-1
Implement CLI command `frontpunch swarm --processes X --concurrency Y --queues Z`.

#### FR-2
Spawning child processes using Python's `multiprocessing` or `subprocess` modules.

#### FR-3
Forward process execution signals (SIGTERM, SIGINT, SIGTSTP) from the parent supervisor to all active children.

#### FR-4
Monitor children and automatically restart child processes that crash (terminate with a non-zero exit code).

### Non-Functional Requirements

#### NFR-1 Performance
- Process monitoring cycles must use non-blocking polls (e.g. check child PIDs every 1 second).

#### NFR-2 Reliability
- The supervisor must handle child exit events cleanly without leaving orphan or zombie processes.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Child processes must inherit parent execution contexts and permissions.

---

## Business Rules

### BR-1
The supervisor parent must only orchestrate; it must never pull or execute jobs directly.

### BR-2
If the supervisor parent receives a SIGTERM/SIGINT, it must forward it, wait for child graceful shutdown cycles to complete, and then exit.

---

## User Flow

### Happy Path

1. Operator runs `frontpunch swarm --processes 2 --concurrency 2`.
2. Supervisor boots, spawns 2 child processes (PID 4401 and 4402).
3. Children register themselves in the Redis process registry and begin executing tasks.
4. Operator runs `kill -TERM <parent_pid>`.
5. Parent forwards SIGTERM to 4401 and 4402.
6. Children exit cleanly; parent exits.

### Alternative Flows

#### AF-1 Child Crash Recovery

1. Swarm is running child 4401 and 4402.
2. Child 4401 encounters OOM and crashes (exit code 137).
3. Supervisor poll detects 4401 is dead.
4. Supervisor prints alert log and spawns a replacement child process (PID 4403).

#### AF-2 Error State

N/A

## Acceptance Criteria

### AC-1 Swarm Process Spawning

**Given**
the swarm command is executed with `--processes 2`

**When**
the boot sequence completes

**Then**
2 child worker processes are spawned, and their PIDs are tracked by the parent process.

---

### AC-2 Signal Forwarding

**Given**
a running swarm containing 2 child workers

**When**
the parent process receives a `SIGTSTP` signal

**Then**
the signal is forwarded to both child processes, placing them in Quiet Mode (US-015).

---

## Edge Cases

### EC-1
A child process exits with code 0 (clean exit).

Expected behavior:
- Assume the child was instructed to shut down and do not restart it (unless shutting down the entire swarm).

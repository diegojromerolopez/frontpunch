## Objective

The worker daemon must support queue weights rather than strictly prioritized polling, preventing queue starvation.

---

## User Story

As a system operator,
I want to specify weights for each queue,
So that worker threads poll queues according to their relative priority weights statistically.

---

## Business Value

- Prevents queue starvation where low priority tasks are never executed.
- Allows balanced distribution of worker resources across queues of varying importance.
- Increases throughput predictability under high queue load.

---

## Context

### Existing Behavior

Queues are prioritized strictly from left-to-right (the left-most queue is fully drained before any jobs are pulled from subsequent queues).

### Desired Behavior

The CLI accepts queue weighting syntax such as `critical:3,default:1`. The worker daemon parses this configuration and uses a weighted random selection algorithm to choose which queue to poll during each fetch cycle.

---

## Requirements

### Functional Requirements

#### FR-1
CLI must accept weighted queue arguments via `--queues critical:3,default:1`.

#### FR-2
Implement a thread-safe random queue selector using the configured weights.

#### FR-3
Worker threads must statistically distribute their retrieval attempts according to the assigned weights.

### Non-Functional Requirements

#### NFR-1 Performance
- Queue selection logic must add less than 0.1ms of overhead per cycle.

#### NFR-2 Reliability
- Starvation must be mathematically bounded based on weight ratios.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

---

## Business Rules

### BR-1
Weights must be positive integers. Decimal or negative weights are disallowed.

### BR-2
If no weight is specified for a queue, it defaults to a weight of 1.

---

## User Flow

### Happy Path

1. Operator runs `frontpunch worker --queues critical:3,default:1`.
2. Daemon boots and calculates probability distribution (75% critical, 25% default).
3. Threads randomly pick a queue using these weights and issue polling fetches.

### Alternative Flows

#### AF-1 Empty State

N/A

#### AF-2 Error State

1. Operator specifies `--queues critical:-1`.
2. System aborts boot and prints invalid weight constraint message.

---

## Acceptance Criteria

### AC-1 Statistical Queue Selection

**Given**
queues are configured with weights `critical:3,default:1`

**When**
threads perform 1000 queue pick selections

**Then**
approximately 750 selections target `critical` and 250 selections target `default` (within a standard margin of error).

---

## Edge Cases

### EC-1
Sum of weights is zero or weight parser fails.

Expected behavior:
- Abort worker startup and print usage error.

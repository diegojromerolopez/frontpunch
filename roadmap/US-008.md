## Objective

Expose a lightweight HTTP REST API (using FastAPI or Flask) to aggregate queue lengths, active workers, and dead-letter tasks, enabling programmatic monitoring.

---

## User Story

As a DevOps engineer,
I want to query a REST API for stats, queues, active workers, and dead retries,
So that I can integrate system metrics into Prometheus/Grafana and retry dead tasks programmatically.

---

## Business Value

- Simplifies operational diagnostics.
- Provides standard endpoints for system integration.
- Reduces MTTR (Mean Time to Resolution) by exposing a simple mechanism to retry failed tasks.

---

## Context

### Existing Behavior

Queue and process states can only be verified by executing raw Redis commands.

### Desired Behavior

Expose a REST API with FastAPI/Flask. `GET /stats` returns job counts. `GET /queues` returns lengths. `GET /workers` parses active process keys. `POST /queues/dead/retry` pops a task from `frontpunch:dead` and inserts it back to its original active queue list.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `GET /stats` returning processed, failed, currently active, and dead counts.

#### FR-2
Implement `GET /queues` returning active queues names and their lengths.

#### FR-3
Implement `GET /workers` returning live worker processes parsed from heartbeat hashes.

#### FR-4
Implement `POST /queues/dead/retry` to retry a job currently in the dead list.

### Non-Functional Requirements

#### NFR-1 Performance
- API response times must be under 50ms.

#### NFR-2 Reliability
- Broker connectivity issues must return clean HTTP 503 Service Unavailable errors.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Endpoint must support basic token authorization rules.

---

## Business Rules

### BR-1
Retrying a dead job must atomically verify its existence in `frontpunch:dead` before moving it, resetting its retry count to 0.

---

## User Flow

### Happy Path

1. Monitor issues `GET /stats`.
2. API queries Redis queue lengths.
3. API returns JSON payload detailing counts.

### Alternative Flows

#### AF-1 Empty State

1. Monitor issues `GET /stats` when no queues are configured.
2. System returns 0 for all properties.

#### AF-2 Error State / Retry missing dead task

1. Operator issues `POST /queues/dead/retry` with invalid `jid`.
2. API searches dead list, finds no match.
3. API returns HTTP 404 Not Found.

---

## Acceptance Criteria

### AC-1 Stats Endpoint

**Given**
the active queue contains 3 tasks and the dead queue contains 1 task

**When**
the user requests `GET /stats`

**Then**
the response returns queue length properties matching these values.

---

### AC-2 Dead Job Retry

**Given**
a task exists in the dead-letter list `frontpunch:dead`

**When**
the user issues `POST /queues/dead/retry` with the task's `jid`

**Then**
the task is enqueued to its active queue, removed from the dead list, and HTTP 200 is returned.

---

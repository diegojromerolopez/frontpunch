## Objective

The daemon must handle process termination signals gracefully by stopping task ingestion, waiting for active threads, and returning uncompleted jobs to Valkey/Redis.

---

## User Story

As a system operator,
I want to terminate workers safely via `SIGTERM` or `SIGINT`,
So that active jobs are completed or safely returned to queues without corrupting execution states.

---

## Business Value

- Protects data integrity during deployments and rolling restarts.
- Eliminates orphaned tasks that would otherwise be lost during abrupt process termination.

---

## Context

### Existing Behavior

Stopping the process via SIGTERM kills running threads abruptly, leaving tasks half-executed and deleted from queue lists.

### Desired Behavior

Upon catching `SIGTERM`/`SIGINT`, the main thread stops pulling new jobs from Redis queues. It waits up to a configurable grace period (e.g. 10 seconds) for running tasks to complete. If any tasks are still running after the grace period, they are cancelled, and the worker pushes their raw payloads back onto the Redis queues (`LPUSH`) before terminating.

---

## Requirements

### Functional Requirements

#### FR-1
Register signal handlers for `SIGTERM` and `SIGINT`.

#### FR-2
Stop thread-loop polling of queue lists immediately upon receiving shutdown signal.

#### FR-3
Implement a timeout-based wait loop for active worker threads (default: 10 seconds).

#### FR-4
Push in-flight/unfinished jobs back onto their respective active queue keys using `LPUSH`.

### Non-Functional Requirements

#### NFR-1 Performance
- Clean shutdown state resolution must not exceed the specified grace period.

#### NFR-2 Reliability
- Safe restoration must prevent race conditions of double-popping when pushing tasks back.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

---

## Business Rules

### BR-1
Jobs pushed back to the queue must have their enqueued timestamp updated to represent the return event.

### BR-2
If the process receives a second SIGTERM/SIGINT, terminate immediately (abrupt termination fallback).

---

## User Flow

### Happy Path

1. Operator issues `kill -15 <pid>`.
2. Worker daemon catches `SIGTERM`, sets `shutdown=True`.
3. Worker stops issuing `BRPOP` commands.
4. Threads finish processing active tasks within 4 seconds.
5. Worker exits cleanly with code 0.

### Alternative Flows

#### AF-1 Grace Period Exceeded

1. Worker daemon catches `SIGTERM`.
2. Active task hangs (external lock or network block).
3. 10 seconds elapse.
4. Worker cancels the executing thread, serializes the in-flight task, pushes it back to Valkey/Redis, and exits.

#### AF-2 Error State

1. Worker daemon catches `SIGTERM` but cannot write to Valkey/Redis to re-queue.
2. System prints raw task details to stderr and exits with non-zero code.

---

## Acceptance Criteria

### AC-1 Interrupted Job Re-queueing

**Given**
a worker daemon is executing a long-running task

**When**
`SIGTERM` is received and the grace period expires before the task completes

**Then**
the task is canceled, its payload is pushed back to `frontpunch:queue:{name}`, and the daemon exits.

---

## Edge Cases

### EC-1
Task is pushed back but the network connection to Redis is severed.

Expected behavior:
- Write warning to stderr containing payload backup for manual recovery.

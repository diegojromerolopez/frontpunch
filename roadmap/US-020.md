## Objective

Prevent job loss when workers crash abruptly (OOM, power failure, SIGKILL) by moving tasks atomically to a worker-specific processing queue before execution.

---

## User Story

As a system operator,
I want to process tasks reliably using a worker-specific in-progress queue list,
So that jobs are never lost if a worker crashes mid-execution.

---

## Business Value

- Guarantees at-least-once task delivery under any sudden process failure.
- Prevents silent failures of critical background jobs (like payment processing or ledger updates).
- Enhances cluster-wide operational reliability and recovery processes.

---

## Context

### Existing Behavior

Workers fetch jobs using `BRPOP` which deletes the job payload from Valkey/Redis immediately. If the worker process is killed before finishing, the job is permanently lost.

### Desired Behavior

Workers fetch jobs using `LMOVE` (or fallback `RPOPLPUSH`) to atomically pop a task from the active queue (`frontpunch:queue:{name}`) and push it to a private, worker-specific processing queue (`frontpunch:processing:{worker_id}:{thread_id}`). Once the task finishes successfully, the worker removes the job from the processing list. If a worker registry key (US-007) expires, a recovery loop scans its processing lists and pushes the orphaned tasks back to their active lists.

---

## Requirements

### Functional Requirements

#### FR-1
Fetch jobs atomically from active queues using Valkey/Redis `LMOVE` (or `RPOPLPUSH`) to move them to a private processing list key `frontpunch:processing:{worker_id}:{thread_id}`.

#### FR-2
Worker threads must delete the job payload from their private processing list immediately upon successful execution.

#### FR-3
Implement a background daemon recovery process that checks process heartbeats (US-007) and identifies orphaned processing queues.

#### FR-4
Orphaned tasks must be atomically pushed back to their original active queue lists (`LPUSH`) for processing by other workers.

### Non-Functional Requirements

#### NFR-1 Performance
- Atomic transfer must add less than 0.5ms of overhead to the fetch cycle.

#### NFR-2 Reliability
- Standby workers must recover orphans without creating duplicate jobs.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

## Business Rules

### BR-1
A job must remain in the worker's private processing queue for the entire duration of its execution.

### BR-2
If the job fails and is scheduled for retry (US-004), it must be removed from the processing queue only after it has been safely written back to the scheduled sorted set.

---

## User Flow

### Happy Path

1. Worker thread calls `LMOVE` to fetch task from `frontpunch:queue:default` to `frontpunch:processing:node1:thread2`.
2. Job executes successfully.
3. Worker deletes job from `frontpunch:processing:node1:thread2`.

### Alternative Flows

#### AF-1 Worker Crash & Recovery

1. Worker A pops job to `frontpunch:processing:nodeA:thread1`.
2. Worker A process is killed abruptly (SIGKILL).
3. Heartbeat key `frontpunch:processes:nodeA` expires.
4. Active Worker B's recovery cycle detects nodeA has expired.
5. Worker B scans `frontpunch:processing:nodeA:*` lists, finds orphaned job, and enqueues it back to `frontpunch:queue:default`.

#### AF-2 Error State

N/A

## Acceptance Criteria

### AC-1 Reliable In-Progress Separation

**Given**
a task exists in `frontpunch:queue:default`

**When**
a worker thread fetches the task

**Then**
the task is moved atomically to `frontpunch:processing:{worker_id}:{thread_id}` and execution begins.

---

### AC-2 Orphan Recovery

**Given**
a task sits in `frontpunch:processing:nodeA:thread1` and nodeA's heartbeat registry key has expired

**When**
the recovery loop executes

**Then**
the task payload is removed from nodeA's list and pushed back onto `frontpunch:queue:default`.

---

## Edge Cases

### EC-1
A network partition isolates Worker A (causing its heartbeat to expire in Redis), but Worker A is still executing the job.

Expected behavior:
- Recovery worker moves the job back to the active queue. When Worker A finishes, its attempt to delete the job from its processing list fails silently. Unique job lock verification (US-010) must be utilized to prevent double-execution when possible.

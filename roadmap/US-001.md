## Objective

Python developers need a clean, decorator-based API to define background tasks and immediately serialize and enqueue them to Valkey/Redis lists for high-throughput messaging.

---

## User Story

As a Python developer,
I want to define tasks using a `@frontpunch.task` decorator and enqueue them using `.delay()` or `frontpunch.enqueue()`,
So that I can offload expensive operations from my request-response cycle.

---

## Business Value

- Simplifies integration for developers adopting the package.
- Establishes a standard, performant serialization format (JSON payload schema) for background jobs.
- Decreases application response times by delegating work asynchronously.

---

## Context

### Existing Behavior

No client API exists to register tasks or send task definitions to a storage broker.

### Desired Behavior

Developers can decorate any Python function with `@frontpunch.task(queue="default", max_retries=5)`. Calling `.delay(*args, **kwargs)` on the decorated function serializes the job payload to a JSON structure and pushes it to a Valkey/Redis List (`frontpunch:queue:{queue_name}`) using `LPUSH`.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `@frontpunch.task(queue="default", max_retries=5)` decorator.

#### FR-2
Implement `task_function.delay(*args)` immediately generating a job payload with a unique `jid` (UUID), `class` (fully qualified path), `args`, `queue`, `created_at` timestamp, and `retry_count`.

#### FR-3
Implement `frontpunch.enqueue(task_fn, *args)` as an alternative enqueueing method.

#### FR-4
All payloads must be serialized to JSON and enqueued to Valkey/Redis list key `frontpunch:queue:{queue_name}` using the Redis client.

### Non-Functional Requirements

#### NFR-1 Performance
- Serialization and enqueuing must complete in under 5ms (excluding network latency).

#### NFR-2 Reliability
- Connection pooling must be used to reuse Valkey/Redis connections.

#### NFR-3 Accessibility
- N/A (Backend API)

#### NFR-4 Security
- Connection URLs must support secure passwords and SSL (`rediss://`).

---

## Business Rules

### BR-1
If a task function is decorated, the original function signature must be preserved (use `functools.wraps`).

### BR-2
Only JSON-serializable arguments are allowed. Non-serializable arguments must raise a `TypeError` during serialization.

---

## User Flow

### Happy Path

1. Developer decorates function with `@frontpunch.task`.
2. Developer calls `my_task.delay(123, "hello")`.
3. System establishes/retrieves connection to Redis, serializes payload to JSON, pushes it to `frontpunch:queue:default`, and returns the `jid`.

### Alternative Flows

#### AF-1 Redis Connection Error

1. Developer calls `.delay()`.
2. Redis connection fails or times out.
3. System raises `frontpunch.exceptions.ConnectionError`.

---

## Acceptance Criteria

### AC-1 Immediate Enqueueing

**Given**
a decorated task `send_email`

**When**
the developer calls `send_email.delay("user@test.com")`

**Then**
a JSON payload is pushed onto `frontpunch:queue:default` and a UUID string `jid` is returned.

---

### AC-2 Serialization Error

**Given**
a decorated task `process_data`

**When**
the developer calls `process_data.delay(object())` (non-serializable object)

**Then**
a `TypeError` is raised.

---

## Edge Cases

### EC-1
Redis server is momentarily partition-blocked.

Expected behavior:
- Retries connection once before raising exception.

## Objective

Developers must be able to inject cross-cutting concerns (logging, metrics, transaction contexts) by registering custom middleware components that intercept job enqueuing and execution.

---

## User Story

As a Python developer,
I want to register client-side and server-side middleware classes,
So that I can log execution details, trace jobs, or manage database transaction lifecycles around jobs globally.

---

## Business Value

- Allows clean integration of monitoring tools (Sentry, OpenTelemetry).
- Reduces code repetition by extracting standard wrappers into reusable modules.
- Promotes modularity in enterprise applications.

---

## Context

### Existing Behavior

Tasks are enqueued and run directly without any wrapping hooks, requiring developers to write logging or transaction wrapper logic inside every task.

### Desired Behavior

Implement middleware registry chains. Client middlewares run when calling `.delay()` (accepting the payload and connection). Server middlewares wrap worker thread executions, accepting the payload and queue, calling `yield` or `next_middleware()` to proceed down the chain.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `frontpunch.client_middleware.add(middleware)` registry.

#### FR-2
Implement `frontpunch.server_middleware.add(middleware)` registry.

#### FR-3
Client middleware must execute sequentially before the client pushes a payload to Valkey/Redis, allowing it to modify the payload or halt enqueuing.

#### FR-4
Server middleware must execute sequentially around job execution in worker threads (supporting the `next` function chain pattern).

### Non-Functional Requirements

#### NFR-1 Performance
- Execution overhead for a simple logging middleware must be under 0.1ms.

#### NFR-2 Reliability
- If a middleware raises an exception, it must propagate safely and follow standard task error retry policies (for server-side middleware).

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Middlewares must not log raw values of encrypted fields (if encrypted job feature is active).

---

## Business Rules

### BR-1
Middlewares must be called in the order they were registered.

### BR-2
If any client middleware raises an exception, the job must not be pushed to Valkey/Redis, and the exception must propagate to the caller.

---

## User Flow

### Happy Path

1. Developer registers `LoggingMiddleware` with `frontpunch.server_middleware.add()`.
2. Job is picked up by a thread.
3. Worker calls the middleware chain.
4. `LoggingMiddleware` prints start details, calls `yield`, task runs, prints completion details.

### Alternative Flows

#### AF-1 Empty State

N/A

#### AF-2 Error State

1. Server middleware raises an exception.
2. The exception is intercepted by the worker retry logic.
3. The job is rescheduled normally.

---

## Acceptance Criteria

### AC-1 Server Middleware Execution

**Given**
a server middleware is registered

**When**
a job executes

**Then**
the middleware's wrapper logic executes around the task run.

---

### AC-2 Chain Interruption

**Given**
a client middleware is registered that intentionally raises a validation exception

**When**
a developer calls `my_task.delay()`

**Then**
the exception is raised to the developer and no job payload is written to Redis.

---

## Edge Cases

### EC-1
Middleware raises an exception inside its cleanup block (post-execution).

Expected behavior:
- Handle the execution as failed and route task to retry state.

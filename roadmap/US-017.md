## Objective

Provide the ability to pause and resume individual queue consumptions dynamically without restarting active worker daemon processes.

---

## User Story

As an administrator,
I want to pause consumption of a specific queue,
So that I can temporarily stop execution of certain types of jobs during system maintenance or outages.

---

## Business Value

- Allows operators to isolate downstream outage impact (e.g. pause `emails` queue when SMTP provider goes down, keeping `billing` processing operational).
- Mitigates retry queue pollution and dead-letter spikes during third-party service downtimes.
- Enhances operator control over database-intensive operations by pacing active queues.

---

## Context

### Existing Behavior

Workers continuously pop items from all configured queues. If a downstream service fails, all corresponding jobs fail and exhaust their retries.

### Desired Behavior

Calling `frontpunch.queue.pause("default")` creates a pause indicator in Redis. The worker thread fetch loop queries this indicator before polling. If a queue is paused, the worker skips polling it and falls back to other non-paused queues. Calling `frontpunch.queue.resume("default")` deletes the indicator, restoring normal consumption.

---

## Requirements

### Functional Requirements

#### FR-1
Implement `frontpunch.queue.pause(queue_name)` and `frontpunch.queue.resume(queue_name)` client methods.

#### FR-2
Store queue pause indicators in Valkey/Redis (`frontpunch:paused:{queue_name}`).

#### FR-3
Worker daemon fetch loops must check the paused state before performing popping executions.

#### FR-4
Expose REST API endpoints `POST /queues/{queue_name}/pause` and `POST /queues/{queue_name}/resume` (usable by the Dashboard).

### Non-Functional Requirements

#### NFR-1 Performance
- Pause checks must not add more than 0.5ms of overhead to worker fetch loops.

#### NFR-2 Reliability
- If Redis connection drops during check, default to treating the queue as unpaused to prevent stall lockups.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- Restrict REST pause/resume endpoints using the standard token authentication rules.

---

## Business Rules

### BR-1
A paused queue does not stop workers from polling other non-paused queues specified in their configuration.

### BR-2
Jobs can still be enqueued (`delay`) into a paused queue; they will accumulate in the broker list until resumed.

---

## User Flow

### Happy Path

1. Operator notices email service outage.
2. Operator triggers `POST /queues/emails/pause`.
3. System creates key `frontpunch:paused:emails` in Redis.
4. Active workers skip `emails` queue and only pop from `default`.
5. Email service restores.
6. Operator triggers `POST /queues/emails/resume` deleting the key.
7. Workers resume processing tasks in `emails`.

## Acceptance Criteria

### AC-1 Queue Pausing

**Given**
queue `default` is paused, and queues contain tasks

**When**
worker polls with configuration `--queues critical,default`

**Then**
the worker only fetches and executes jobs from the `critical` queue.

---

### AC-2 Queue Resuming

**Given**
queue `default` is currently paused

**When**
the queue is resumed and new tasks are enqueued

**Then**
the worker immediately begins polling and executing tasks from the `default` queue.

---

## Edge Cases

### EC-1
A worker thread is already blocking on `BRPOP` for a queue when it is paused.

Expected behavior:
- The worker will process the single job that pops out of that blocking call, but subsequent fetch loops will skip the queue.

## API Changes

### Endpoints

#### POST /queues/{queue_name}/pause
- Request: Empty
- Response: `{"status": "paused", "queue": "default"}`

#### POST /queues/{queue_name}/resume
- Request: Empty
- Response: `{"status": "active", "queue": "default"}`

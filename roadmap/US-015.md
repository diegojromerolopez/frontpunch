## Objective

Support zero-downtime worker redeployments by handling termination pre-signals that stop task consumption but allow existing processing threads to finish cleanly.

---

## User Story

As an operations engineer,
I want to quiet workers via the `SIGTSTP` signal before shutting them down,
So that I can stop them from taking new jobs while they complete their active workload during deployment rolling restarts.

---

## Business Value

- Decreases task aborts and retries during worker redeployments.
- Provides operations with a simple, standard mechanism to phase out active worker nodes safely.
- Eliminates execution overlaps during rolling zero-downtime updates.

---

## Context

### Existing Behavior

Workers only respond to SIGTERM/SIGINT. If stopped, they either terminate immediately or try to push running tasks back to the queue after a grace timeout.

### Desired Behavior

Upon receiving a `SIGTSTP` signal, the worker enters "Quiet Mode". It immediately ceases pulling new jobs from active queues. However, the worker process remains alive, letting active thread pools execute current jobs to completion. Once active threads reach 0, the operator can safely issue `SIGTERM` to exit the worker process instantly.

---

## Requirements

### Functional Requirements

#### FR-1
Register signal handlers for `SIGTSTP` (terminal suspend).

#### FR-2
Setting a `quiet=True` state flag on signal intercept, stopping worker loop from performing `BRPOP`.

#### FR-3
Process heartbeat metadata (US-007) must be updated to report `quiet: true` state to Redis.

### Non-Functional Requirements

#### NFR-1 Performance
- Transitioning to quiet mode must take under 1ms.

#### NFR-2 Reliability
- Active thread states must be tracked safely.

#### NFR-3 Accessibility
- N/A

#### NFR-4 Security
- N/A

---

## Business Rules

### BR-1
Quiet mode is a one-way state transition. Once quieted, a worker cannot resume polling active queues; it must be restarted.

### BR-2
If the worker is quieted and receives `SIGTERM` or `SIGINT`, it must trigger the graceful/abrupt shutdown workflow immediately.

---

## User Flow

### Happy Path

1. Operator issues `kill -TSTP <pid>`.
2. Worker daemon catches `SIGTSTP`, sets `quiet=True` and updates heartbeat status in Redis.
3. Thread loops stop fetching new tasks.
4. Active tasks complete running (e.g. 5 seconds).
5. Active threads count drops to 0.
6. Operator issues `kill -15 <pid>` (SIGTERM).
7. Worker exits cleanly with code 0 since no tasks are active.

## Acceptance Criteria

### AC-1 Quiet Mode Transition

**Given**
a worker process is running and executing 2 active tasks

**When**
`SIGTSTP` signal is received

**Then**
the worker stops fetching new tasks, let the 2 active tasks complete, and the worker process remains alive.

---

## Edge Cases

### EC-1
Worker process is quieted but never receives SIGTERM, staying idle forever.

Expected behavior:
- Keep worker process alive as an idle process. Heartbeats will continue updating its `quiet: true` state in Redis.
